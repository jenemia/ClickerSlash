# 레인 구역 구현 문서

## 목적
- 이 문서는 프로토타입 전투의 기본 작업 공간인 레인 구역 구현 방식과 현재 미니게임 구조를 정리한다.
- 기준 구현은 `PrototypeBattle` 씬과 ECS 전투 루프다.

## 레인 구역 역할
- 플레이어가 좌우로 레인을 이동하며 내려오는 물류를 처리하는 메인 작업 공간이다.
- 물류를 성공 처리하면 돈과 콤보가 올라가고, 같은 물류 타입이 상하차 큐에도 적재된다.
- 상하차 구역으로 이동하지 않아도 레인 작업은 계속 진행된다.

## 공간 구성
- 전장은 4개 고정 레인으로 구성된다.
- 상단은 물류 스폰 라인, 하단은 판정선과 실패선, 최하단은 플레이어 위치다.
- 카메라는 레인 전체와 하단 방어 구역을 동시에 보는 고정 시점이다.

## 핵심 런타임 구조
- `BattleConfig`
  - 스폰 간격, 판정선, 실패선, 이동 시간 같은 전투 규칙을 가진다.
- `PlayerTag`, `LaneIndex`, `LaneMoveState`
  - 플레이어가 어느 레인에 있는지와 이동 중 상태를 표현한다.
- `CargoTag`, `CargoKind`, `CargoWeight`, `CargoReward`, `CargoPenalty`
  - 레인에서 내려오는 물류 엔티티의 속성을 가진다.
- `BattleSessionStatsState`, `ComboState`
  - 현재 세션의 돈, 처리 수, 콤보를 기록한다.

## 미니게임 구조
- 입력
  - `A / D` 또는 `Left / Right`로 한 칸씩 레인을 이동한다.
  - 입력 예약이 가능하며, 큐에 쌓인 이동 명령은 순차 처리된다.
- 판정
  - 플레이어와 같은 레인에 들어온 물류가 판정선 근처에 왔을 때 처리된다.
  - 무게가 최대 처리 가능 무게 이하이면 성공, 초과하면 실패 처리된다.
- 성공 결과
  - 보상 획득
  - 콤보 증가
  - `CargoHandledEvent` 발생
  - 같은 `CargoKind`가 상하차 세션 큐에 적재
- 실패 결과
  - 패널티 적용
  - 콤보 초기화
  - `CargoMissedEvent` 발생

## 관련 시스템 흐름
- `CargoSpawnSystem`
  - 랜덤 레인에 물류를 스폰하고 `CargoKind`를 부여한다.
- `PlayerInputCollectSystem`
  - 좌우 입력을 `LaneMoveCommandBufferElement`로 수집한다.
- `PlayerLaneQueueSystem`
  - 이동 큐에서 다음 이동을 꺼내 실제 레인 이동 상태로 전환한다.
- `PlayerLaneMoveSystem`
  - 현재 목표 레인으로 보간 이동한다.
- 처리/판정 시스템
  - 물류 성공 또는 실패를 판정하고 세션 통계를 갱신한다.

## 상하차와의 연결
- 레인에서 성공 처리된 물류만 상하차 큐에 적재된다.
- 적재는 도크 진입 여부와 상관없이 즉시 일어난다.
- 플레이어가 상하차 구역에 있는 동안에는 레인 이동 입력과 이동 큐가 잠긴다.
- 하지만 레인 물류 스폰과 처리 결과는 세션 기준으로 계속 누적된다.

## 시각 표현
- 레인 물류는 `Standard / Fragile / Heavy` 타입별 공유 프리팹을 사용한다.
- 같은 타입은 상하차 구역에서도 동일한 프리팹으로 보인다.
- 플레이어와 물류의 시각 오브젝트는 ECS 엔티티를 읽는 브리지에서 생성한다.

## 현재 주의점
- 레인 작업은 물류 처리 루프의 중심이고, 상하차는 보조 작업 공간이다.
- 상하차 진입 중에는 레인 플레이어 이동을 막지만, 세션 자체는 계속 흐른다.
- 레인 카메라 구도와 도크 가시성은 별도 연출 문제이며, 본 문서는 레인 미니게임 구조만 다룬다.
