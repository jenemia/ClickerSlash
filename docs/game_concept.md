# 게임 컨셉 문서

## 게임 한 줄 소개
- **3D로 표현된 4개의 레인에서 적이 상단에서 내려오고, 플레이어가 하단에서 좌우 이동과 자동공격으로 막아내며 전투 후 스킬 트리 성장으로 방어 운영 방식을 확장하는 PC/Web 가로형 반복 전투 게임**

## 핵심 재미
| 축 | 설명 | 플레이 감각 |
| --- | --- | --- |
| 레인 읽기 | 어떤 레인에 먼저 가야 하단 돌파를 막을 수 있는지 빠르게 해석 | 짧은 순간 판단 |
| 자리 선점 | 이동 중 공격이 끊기기 때문에 어느 레인을 먼저 점유할지 결정해야 함 | 운영형 긴장감 |
| 자동공격 유지 | 도착 즉시 자동공격이 이어지므로 자리를 잘 잡을수록 안정적으로 정리 가능 | 리듬감 있는 반복 템포 |
| 3D 전장 가독성 | 입체감이 있지만 카메라와 레인 연출이 단순해 위협 위치를 빠르게 읽을 수 있음 | 손에 잡히는 현장감 |
| 짧은 반복 전투 | 60초 생존전이라 실패 부담이 낮고 즉시 재도전 가능 | 속도감 있는 반복 |
| 누적 성장 | 전투 결과가 다음 전투의 대응 방식으로 되돌아옴 | 방치형/클리커형 축적 만족 |
| 빌드 차별화 | 스킬 트리로 이동 보조, 공격 범위 확장, 자동화 강화 등 플레이 양상이 달라짐 | 장기 목표 형성 |

## 타깃 유저
| 유저군 | 기대 포인트 | 대응 설계 |
| --- | --- | --- |
| 운영형 액션 라이트 유저 | 복잡한 조작 없이도 빠른 상황 판단을 즐기고 싶음 | 4레인 선점과 하단 돌파 저지 중심 설계 |
| 방치형/클리커 유저 | 조금씩 누적되는 성장, 다음 판이 더 편해지는 구조 선호 | 영구 성장, 오프라인 보상, 해금 트리 제공 |
| 짧은 세션 PC/Web 유저 | 1~2분 안에 한 판 완료 가능해야 함 | 전투 60초 목표 |
| 빌드 최적화 유저 | 같은 전투라도 다른 방식으로 공략하고 싶음 | 수치보다 플레이 스타일 변조형 노드 제공 |

## 플레이 루프
1. 메타 화면에서 스킬 트리, 장착 특성, 스테이지를 선택한다.
2. 전투에 진입해 4레인에 랜덤하게 등장하는 적 흐름을 보고 지켜야 할 레인을 판단한다.
3. 좌우 이동으로 레인을 선점하고 자동공격을 유지해 적의 하단 돌파를 막는다.
4. 콤보를 유지하며 더 안정적으로 적을 정리하고 전투를 생존한다.
5. 전투 종료 후 재화, 조각, 해금 포인트를 획득한다.
6. 메타 성장 화면에서 신규 노드 개방, 플레이스타일 전환, 오프라인 효율 상승을 적용한다.

## 차별점
| 비교 대상 | 일반 구조 | 본 프로젝트 구조 |
| --- | --- | --- |
| 일반 PC/Web 액션 | 이동 + 공격 + 회피 다중 입력 | 좌우 이동과 자동공격 유지에 초점 압축 |
| 전형적 오토배틀 | 자동 배치 후 관전 위주 | 플레이어가 직접 레인을 옮겨 방어 우선순위를 계속 선택 |
| 일반 방치형 RPG | 수치 성장으로 전투를 자동 통과 | 성장으로 플레이 방식을 수정하되, 레인 운영 선택은 유지 |
| 라인 디펜스류 | 적 흐름 관리 중심 | 플레이어가 직접 레인을 점유해 “몸으로” 막는 구조 |

## 3D 전장 방향
| 항목 | 방향 |
| --- | --- |
| 카메라 | 가로형 화면에 맞춘 고정 원근 카메라, 플레이어와 하단 방어선이 항상 보이게 유지 |
| 레인 표현 | 바닥 스트립, 발광 라인, 레인별 경계 메쉬로 구분 |
| 적 표현 | 실루엣과 머티리얼 대비로 속도/체력 차이를 즉시 식별 가능하게 설계 |
| 연출 우선순위 | 화려함보다 읽기 쉬움 우선, 이펙트는 레인 식별을 방해하지 않게 제한 |
| 플레이 감각 | 2D UI처럼 명확하지만 3D 장난감 전장 같은 현장감을 제공 |

## 왜 이 게임이 반복 플레이에 적합한가
| 반복 요인 | 이유 |
| --- | --- |
| 짧은 세션 | 실패 후 즉시 재도전 가능해 반복 허들이 낮다. |
| 하단 돌파 압박 | 같은 스폰 구조도 레인 우선순위 선택에 따라 결과가 크게 달라진다. |
| 콤보 욕심 | 같은 스테이지도 하단 통과 없이 더 오래 콤보를 유지하고 싶어진다. |
| 성장 체감 | 한 판의 보상이 다음 판의 이동 안정성이나 공격 범위 확장으로 이어진다. |
| 빌드 실험 | 이동 보조형, 공격 확장형, 자동화형 등 트리 방향이 다르면 같은 스테이지도 감각이 달라진다. |

## 전투와 성장의 결합 원칙
- 전투는 실력만 보는 구조가 아니라, 메타 성장으로 **이동 여유**, **콤보 유지성**, **공격 범위 확장 가능성**을 해금하는 구조여야 한다.
- 성장으로 모든 입력을 없애면 안 되고, 성장 전후의 차이는 “편해졌다”보다 “다른 방식으로 풀 수 있다”에 가깝게 설계한다.
- ECS 기준으로 보면 메타 성장 결과는 전투 진입 시점의 `SkillLoadoutState`, `AutoAttackProfile`, `RewardModifier` 등 초기 컴포넌트 집합으로 주입되어야 한다.

## 세션 길이와 콘텐츠 소비 속도
| 구간 | 목표 시간 | 목적 |
| --- | --- | --- |
| 튜토리얼 전투 | 30~45초 | 좌우 이동과 자동공격 인지 |
| 일반 스테이지 | 60초 | 반복 플레이 기본 단위 |
| 보스 스테이지 | 75~90초 | 장기 운영과 빌드 검증 |
| 메타 정비 | 30~90초 | 성장 선택과 재도전 동기 형성 |

## 서비스 확장성 메모
- 시즌제 콘텐츠를 붙이기 쉽다. 스폰 규칙, 적 프리셋, 시즌 트리만 추가해도 변형이 가능하다.
- 스테이지와 성장 노드가 데이터 중심이면 DOTS 런타임은 유지한 채 콘텐츠 확장이 가능하다.
- 첫 버전은 단일 캐릭터로 시작하되, 후속 버전에서 캐릭터별 공격 커버 범위나 이동 특성 추가가 가능하다.
- 3D 전장 자산은 스킨, 테마, 배경 세트 확장에 유리하되, 카메라와 레인 가독성 규칙은 공통으로 유지해야 한다.

## 실행 우선순위
1. 게임의 정체성을 “4레인 자동공격 생존전”으로 명확히 고정한다.
2. 전투 1판 길이를 60초 기준으로 먼저 맞춘다.
3. 메타 성장은 적어도 3개 플레이 스타일 분기를 가진 트리로 설계한다.
4. 수치 성장보다 규칙 변화형 노드를 먼저 정의한다.
